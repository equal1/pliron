# pliron Python bindings — design documentation

This directory documents the design of pliron's Python bindings as implemented in
the "first phase" commit and the subsequent core/bindings split refactor. It is
written to serve as a **saved spec**: a faithful description of how Rust IR is
exposed to Python, so the design can be extended consistently in later phases.

The bindings are built with [PyO3](https://pyo3.rs) and packaged with
[maturin](https://www.maturin.rs). They have two halves:

- **Hand-written core**, at the crate root of `pliron-python` — wrappers
  for the fixed IR entities (`Context`, `Operation`, `BasicBlock`, `Region`,
  `Value`, `Type`, `Attribute`) plus the builder infrastructure.
- **Generated dialect surface**, emitted by the `pliron-python-derive`
  proc-macros — one `#[pyclass]` per dialect type / attribute / op, produced
  from the same `#[pliron_type]` / `#[pliron_attr]` / `#[pliron_op]`
  definitions that already drive the Rust side. The core `pliron` crate stays
  entirely Python-free: its derive macros only export each annotated item's
  tokens as an inert "reflect" macro, which the Python codegen consumes.

Both halves live in the `pliron-python` crate, which is compiled (as a
`cdylib`) into the actual CPython extension module and doubles as a library
(`rlib`) for downstream dialect crates.

## One-paragraph summary

A Python program opens a `with pliron.Context():` block, which installs a raw
pointer to the Rust `Context` in a thread-local. Every Python IR handle is a thin
wrapper around an arena pointer (`Ptr<Operation>`, `Ptr<TypeObj>`, …) or a boxed
value (`AttrObj`); methods resolve their data by reading that thread-local
context (`get_ctx()`), so handles carry no lifetime and no borrow. Dialect
classes are generated by the `pliron-python-derive` macros: each
`#[pliron_type]`/`_attr`/`_op` exports its item tokens as a Python-agnostic
reflect macro, and a bindings crate turns each export into a `Py<Name>`
`#[pyclass]` (same-crate items can instead stack the `#[py_type]`/`#[py_attr]`/
`#[py_op]` attributes directly), with field accessors whose Rust↔Python
conversions are resolved through the `PyMap` trait. Generated classes append
themselves to a global `linkme` slice
(`pliron_python::statics::PY_CLASS_REGISTRATIONS`), which `pliron-python`'s
`#[pymodule]` walks at import time to register everything.
