# pliron Python bindings â€” design documentation

This directory documents the design of pliron's Python bindings as implemented in
the "first phase" commit (`bb7425d`). It is written to serve as a **saved spec**:
a faithful description of how Rust IR is exposed to Python, so the design can be
extended consistently in later phases.

The bindings are built with [PyO3](https://pyo3.rs) and packaged with
[maturin](https://www.maturin.rs). They have two halves:

- **Hand-written core**, in `src/python/` of the core `pliron` crate â€” wrappers
  for the fixed IR entities (`Context`, `Operation`, `BasicBlock`, `Region`,
  `Value`, `Type`, `Attribute`) plus the builder/rewriter infrastructure.
- **Generated dialect surface**, emitted by the `pliron-derive` proc-macros â€”
  one `#[pyclass]` per dialect type / attribute / op, produced from the same
  `#[pliron_type]` / `#[pliron_attr]` / `#[pliron_op]` definitions that already
  drive the Rust side.

The whole thing is glued together by the thin `pliron-python` assembly crate,
which is the actual compiled CPython extension module.

## One-paragraph summary

A Python program opens a `with pliron.Context():` block, which installs a raw
pointer to the Rust `Context` in a thread-local. Every Python IR handle is a thin
wrapper around an arena pointer (`Ptr<Operation>`, `Ptr<TypeObj>`, â€¦) or a boxed
value (`AttrObj`); methods resolve their data by reading that thread-local
context (`get_ctx()`), so handles carry no lifetime and no borrow. Dialect
classes are generated by the derive macros: each `#[pliron_type]`/`_attr`/`_op`
emits a `Py<Name>` `#[pyclass]` gated behind `#[cfg(feature = "python")]`, with
field accessors whose Rustâ†”Python conversions are resolved through the `PyMap`
trait. Generated classes append themselves to a global `linkme` slice, which the
assembly crate's `#[pymodule]` walks at import time to register everything.
